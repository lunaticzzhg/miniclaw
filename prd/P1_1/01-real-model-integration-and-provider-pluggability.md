# P1_1 真实模型链路与模型可插拔

## 目标
在 P1 本地假数据或简化链路的基础上，跑通真实大模型请求链路，并建立可插拔的模型 Provider 接入能力，为后续 P2 Agent 工具调用闭环提供稳定底座。

## 阶段定位
- P1 解决“聊天闭环是否成立”
- P1_1 解决“聊天背后是否是真实模型，以及模型接入是否可扩展”
- P2 再在真实模型基础上继续增加 Agent 推理与工具调用闭环

说明：
- P1_1 不替代 P1，而是在 P1 与 P2 之间增加一层真实能力验证
- P1_1 以 MiniMax 作为首个试点模型服务
- 从产品层面要求模型能力可插拔，避免后续被单一厂商绑定

## 用户可获得的价值
- 用户在 App 内发出的消息，能得到真实模型返回的回答
- 用户可以感知当前是否已经连接到可用模型服务
- 当模型服务异常时，用户可以明确感知失败原因，而不是只看到“无响应”
- 后续产品升级模型服务时，用户侧核心聊天体验不需要被推翻重做

## 为什么要做这一期
- 如果没有真实模型链路，P1 的聊天体验只能验证页面和本地状态，无法验证真实产品闭环
- 如果在 P2 才首次接入真实模型，会把“模型接入问题”和“Agent/工具调用问题”混在一起，排查成本高
- 如果不提前做 Provider 抽象，后续接入不同模型时容易把供应商细节渗透到业务层和页面层

## 核心设计原则
- 先保证“真实可用”，再追求“多供应商扩展”
- Provider 差异必须收敛在接入层，不能污染页面交互和业务语义
- 用户不需要理解底层供应商概念，但需要知道当前模型是否可用
- 异常优先局部化呈现，不因为模型失败就破坏整个会话容器
- 配置与密钥管理要为开发调试留入口，但不打断普通聊天流程

## 本期范围
- 接入至少 1 个真实模型服务，首期为 MiniMax
- App 内聊天请求由真实模型返回，不再依赖纯本地假回复
- 支持基础流式或准流式回复体验，保持与 P1 聊天状态设计一致
- 建立统一的模型 Provider 抽象层
- 支持在配置层切换默认模型 Provider
- 增加模型可用性校验与失败反馈
- 为后续扩展 OpenAI、OpenRouter、Claude 或自建网关预留接入位

## 非目标
- 本期不实现多模型同时在线切换的复杂用户界面
- 本期不做模型市场、模型广场或高级参数调优面板
- 本期不引入工具调用、函数调用、Agent 规划能力
- 本期不解决多端统一配置同步
- 本期不做复杂的计费、额度、套餐管理

## 试点方案
### MiniMax 作为首个 Provider
- 选择 MiniMax 作为真实模型试点
- 优先验证文本对话主链路是否稳定可用
- 优先验证 Android 侧流式接收、超时处理、取消回复和失败重试是否成立

### 可插拔要求
- 产品能力上不写死为“MiniMax 聊天”
- 所有页面文案、状态定义、数据对象都以“模型服务”或“AI 服务”为抽象
- MiniMax 只是默认接入实现，而不是架构唯一前提

## 用户场景
### 场景 1：真实聊天验证
1. 用户打开 App 并进入聊天页
2. 用户发送一条文本消息
3. 系统将消息提交给真实模型服务
4. 用户看到 AI 真实返回的内容，并按流式过程逐步展示

### 场景 2：模型服务不可用
1. 用户发送消息
2. 当前模型服务鉴权失败、超时或网络异常
3. 对应消息展示失败状态和可重试入口
4. 用户可以在修复配置后再次发送或重试

### 场景 3：后续切换 Provider
1. 产品侧新增另一个模型 Provider
2. 不需要重做聊天页、会话页和消息状态机
3. 新 Provider 可以复用现有发送、流式、失败、重试和停止回复体验

## 产品要求
### 一、真实模型链路要求
- 用户消息必须实际发往真实模型服务
- assistant 回复必须来自真实模型返回结果
- 回复过程应尽量保持 P1 已定义的 `思考中 -> 回复中 -> 已完成 / 失败 / 已停止`
- 用户点击停止后，应尽快中断当前模型返回

### 二、Provider 抽象要求
- 产品层只认统一的聊天能力，不感知供应商专有字段
- 不同 Provider 的鉴权方式、请求格式、响应格式必须在接入层被适配
- 上层会话、消息、状态机、重试逻辑不因更换 Provider 而重写

### 三、配置要求
- 开发阶段必须存在最小可用配置方式，用于填写或读取 MiniMax 所需配置
- 配置缺失时，应给出明确提示，不能表现为静默失败
- 配置错误时，应能区分“未配置”“鉴权失败”“网络失败”“服务异常”等常见问题

### 四、失败反馈要求
- 用户发送失败时，失败信息优先贴近对应消息展示
- 若当前 App 尚未配置有效模型服务，可提供一次性轻提示或引导
- 不要求在本期设计复杂设置中心，但必须有能支撑开发与演示的配置入口

## 交互设计
### 页面影响范围
P1_1 仍基于 P1 的两个核心页面：
- 会话列表页
- 聊天页

本期原则上不新增复杂业务页面，只在必要时补充最小配置入口或状态提示。

### 聊天页新增感知点
- 当模型请求已发出但尚未收到首包时，保持“思考中”
- 当收到真实流式内容后，进入“回复中”
- 当由于配置缺失无法请求模型时，在消息附近给出明确失败反馈
- 当用户主动停止时，保留已生成内容并标记“已停止”

### 可能新增的轻量配置入口
可接受以下任一形式：
- 启动时检测未配置，进入一个极简配置引导
- 在会话列表或聊天页提供开发期配置入口
- 通过调试构建或开发设置页完成配置

要求：
- 配置入口不能喧宾夺主
- 普通聊天主路径仍应保持简洁
- 一旦配置完成，用户应能快速回到聊天流程

## 成功标准
- MiniMax 链路在 Android App 内真实可用
- 用户可连续进行多轮真实对话
- 流式展示、停止回复、失败重试三条主链路在真实模型下成立
- 切换或扩展 Provider 时，不需要修改页面级交互定义
- P2 在此基础上可以直接叠加 Agent 和工具调用能力，而不需要回头重构模型接入层

## 验收标准
- 至少 1 个真实模型 Provider 接入成功，默认使用 MiniMax
- 用户在 App 内发送消息后，可收到真实模型回复
- 连续发送 5 轮消息时，真实上下文连续有效
- 模型请求过程中，界面仍可明确区分“思考中”“回复中”“已完成”
- 模型服务异常时，用户可看到失败状态并执行重试
- 用户可在回复过程中点击停止，并保留已生成内容
- 新增第二个 Provider 时，不需要重做聊天页交互与消息状态设计

## 测试方式
1. 配置有效的 MiniMax 凭据，发送普通问题，验证真实回复成功返回
2. 连续发送 5 轮上下文相关问题，验证模型具备连续上下文表现
3. 请求长回答，验证流式或准流式展示符合预期
4. 回复生成中点击停止，验证模型返回被中断且内容保留
5. 配置错误凭据，验证界面能明确反馈鉴权失败
6. 断网后发送消息，验证失败提示与重试链路成立
7. 将默认 Provider 切换为另一实现或 Mock 实现，验证聊天主页面不需要重新设计

## 对后续 P2 的衔接要求
- P2 的 Agent 推理必须建立在真实模型可用前提下
- P2 若需要函数调用或工具调用扩展，应复用本期建立的 Provider 抽象边界
- P2 新增的 tool call / tool result 消息类型，不应破坏本期已建立的真实聊天主链路

## 风险与注意事项
- MiniMax 接口能力、流式协议和鉴权规则可能与其他供应商不同，必须避免把这些差异上抬到业务层
- 如果本期只验证成功链路而忽略失败链路，P2 叠加复杂能力后问题会被放大
- 如果本期不明确配置方案，开发与测试阶段会频繁被环境问题阻塞
- 如果本期不做可插拔抽象，后续接入第二家模型服务的成本会明显升高
