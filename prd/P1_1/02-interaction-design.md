# P1_1 交互设计文档

## 文档目标
明确 P1_1「真实模型链路与模型可插拔」的页面结构、核心交互、状态反馈与边界场景，作为后续 UI 设计、Android 实现与验收对齐依据。

## 设计目标
- 在不破坏 P1 聊天主路径的前提下，让用户真正使用到真实模型能力
- 用户可以明确感知当前模型服务是否可用，而不是在失败后才猜测问题
- 当模型配置缺失、鉴权失败、网络异常时，用户能在原上下文中理解问题并恢复
- 模型 Provider 的切换或扩展，不改变用户对聊天主流程的理解

## 本期页面范围
- 会话列表页
- 聊天页
- 模型配置引导页或模型配置面板

说明：
- P1_1 仍以会话列表页和聊天页为主
- 新增页面只服务“真实模型链路可用性”这一目标，不扩展成完整设置中心

## 核心交互原则
- 聊天仍然是主路径，配置只是为让聊天可用
- 模型切换入口仅保留在聊天页，不在会话列表页分散注意力
- 配置问题优先在用户即将受阻时暴露，不做过早打扰
- 所有模型相关异步过程都必须有清晰状态反馈
- Provider 差异不暴露给普通聊天交互
- 失败必须尽量原位恢复，避免把用户踢出当前会话

## 信息架构
### 一级结构
- 会话列表页：查看已有会话、进入会话、创建新会话
- 聊天页：发送消息、查看真实模型回复、处理失败与重试
- 模型配置引导页/面板：在模型不可用时完成最小配置

### 对象层级
- 会话：连续对话容器
- 消息：用户与 assistant 的单条内容及状态
- 模型 Provider：当前承载 AI 回复能力的服务提供者，例如 MiniMax
- 模型可用性：配置是否完整、鉴权是否成功、当前是否可发送请求

## 页面一：会话列表页
### 页面目标
- 保持 P1 已建立的默认入口体验
- 让用户快速进入已有会话或新建会话

### 页面组成
- 顶部区域：产品标题
- 会话列表区域：展示会话卡片
- 空态区域：无会话时展示引导文案
- 主操作按钮：新建会话

### 交互规则
1. 用户点击某个会话卡片，进入对应聊天页
2. 用户点击“新建会话”，创建新会话并进入聊天页

### 状态设计
- 默认态：展示会话列表
- 空态：展示“还没有会话，开始第一段对话”

## 页面二：聊天页
### 页面目标
- 让用户在熟悉的聊天界面内使用真实模型能力
- 让用户理解“正在请求真实模型”这一过程
- 在配置缺失、鉴权失败、网络异常等场景下保留上下文并给出恢复路径

### 页面组成
- 顶部栏：返回按钮、会话标题、右侧模型切换入口
- 消息流区域：按时间顺序展示用户与 AI 消息
- 底部输入区：输入框、发送按钮、停止按钮
- 可选局部提示区：当模型不可用时，展示轻量说明或配置入口

### 顶部栏规则
- 默认展示会话标题
- 顶部栏最右侧固定提供模型切换入口
- 模型切换入口可展示当前 Provider 名称、简称或状态标识
- 对普通用户，不要求理解 Provider 细节
- 用户点击该入口后，可进入模型配置引导页或配置面板
- 对开发/测试场景，可通过该入口完成 Provider 切换与配置修复
- 当 assistant 正在回复时，仍允许查看当前模型信息，但不建议在该时刻直接切换并影响进行中的请求

### 消息流设计
- 延续 P1 的用户消息与 AI 消息展示方式
- assistant 消息仍按“思考中 -> 回复中 -> 已完成 / 回复失败 / 已停止”演进
- 若失败原因为模型配置缺失、鉴权失败、网络失败，失败信息应尽量贴近对应 assistant 消息
- 错误文案要帮助用户区分“当前消息失败”和“整个会话不可继续”的区别

### 输入区设计
- 输入框继续支持多行输入
- 输入内容为空时，发送按钮不可用
- 点击发送后，用户消息立即进入消息流
- 当请求已经发出且 assistant 正在生成时，发送按钮切换为停止按钮
- 若当前模型明确不可用，点击发送后不应静默无响应，应进入明确失败反馈或先引导配置

## 页面三：模型配置引导页或面板
### 页面目标
- 在不建立复杂设置系统的前提下，提供真实模型链路所需的最小配置能力
- 帮助用户或开发者快速完成 MiniMax 试点配置
- 在配置完成后，尽快回到聊天主流程

### 使用时机
- 首次尝试发送消息时发现尚未配置模型
- 用户主动点击顶部栏右侧的模型切换入口
- 鉴权失败或配置失效后，用户希望修复问题

### 页面组成
- 标题区域：说明这是模型配置，而不是完整设置中心
- Provider 选择区域：默认选中 MiniMax，可为后续 Provider 预留位置
- 配置输入区域：填写当前 Provider 的必要参数
- 测试连接或保存按钮
- 完成后返回聊天的主操作

### Provider 选择设计
- 模型切换入口进入后，用户可以看到当前已选 Provider
- 若当前阶段只接入 MiniMax，也应保留单选展示形式，为后续扩展保持一致体验
- 当后续接入多个 Provider 时，用户可在此处切换默认 Provider
- 切换行为应被视为配置行为的一部分，与参数编辑、测试连接放在同一交互闭环内

### 交互规则
1. 首次未配置时，用户可被引导进入该页或面板
2. 默认只展示当前阶段必须填写的最少字段
3. 用户点击保存或测试连接后，页面反馈“成功 / 失败”
4. 若成功，用户应能一键返回原会话继续发送
5. 若失败，错误信息应尽量明确，不只显示“请求失败”

### 状态设计
- 未配置：提示需要完成模型配置
- 编辑中：允许修改配置项
- 校验中：点击测试连接或保存后展示进行中状态
- 配置成功：提示已可开始真实对话
- 配置失败：展示原因并允许继续修改

## 聊天核心流程
### 流程 1：模型已配置，正常真实对话
1. 用户进入聊天页
2. 用户输入消息并点击发送
3. 用户消息立即进入消息流
4. 系统向真实模型 Provider 发起请求
5. assistant 消息进入“思考中”
6. 收到首包后进入“回复中”
7. 回复结束后进入“已完成”

### 流程 2：首次发送时发现未配置模型
1. 用户进入聊天页并输入消息
2. 用户点击发送
3. 系统检测当前无有效模型配置
4. 页面在原上下文中提示“需要先完成模型配置”
5. 用户进入配置引导页或配置面板
6. 配置成功后返回聊天页
7. 用户重新发送消息或继续当前输入

### 流程 3：模型鉴权失败
1. 用户发送消息
2. 请求到达模型服务，但鉴权失败
3. 对应消息显示失败状态
4. 失败文案明确提示“配置无效”或“鉴权失败”
5. 用户可选择重试或前往修复配置

### 流程 4：网络异常或服务超时
1. 用户发送消息
2. 模型请求超时或网络中断
3. assistant 消息显示失败状态
4. 用户可直接重试
5. 若多次失败，可补充轻提示建议检查网络或服务状态

### 流程 5：回复途中停止
1. assistant 正在回复中
2. 输入区展示停止按钮
3. 用户点击停止
4. 当前请求尽快取消
5. 已生成内容保留，状态变为“已停止”

## 模型状态设计
### 状态分类
- 已配置且可用
- 未配置
- 配置中
- 鉴权失败
- 网络异常
- 服务暂不可用

### 表现原则
- “可不可用”是模型服务状态，不应混同于单条消息状态
- 页面级模型状态用于辅助理解，消息级状态用于表达本轮请求结果
- 同一问题优先在离用户操作最近的位置提示

### 建议表现位置
- 顶部栏最右侧模型切换入口：表达当前模型状态总览，并承担切换入口职责
- 消息项局部状态：表达单轮请求成功或失败
- 配置页反馈区：表达当前配置是否有效

## 失败与恢复设计
### 配置缺失
- 不建议用户发送后只看到空白等待
- 应尽快提示“需要先配置模型”
- 提供直接进入配置的入口

### 鉴权失败
- 失败信息展示在对应请求上下文附近
- 用户可看到“重试”和“修复配置”两个方向
- 若用户未修复配置，重复发送不应伪装成正常处理中

### 网络失败
- 优先原位展示“网络异常，可重试”
- 不要求复杂全局弹窗
- 若用户返回后重新进入，历史消息仍应可见

### Provider 切换失败
- 用户通过顶部栏模型切换入口进入后，若切换失败，应在配置页处理
- 不应直接污染当前已存在的聊天消息流

## 文案与反馈建议
### 状态文案
- 发送中
- 思考中
- 回复中
- 回复失败，可重试
- 已停止
- 模型未配置
- 配置无效，请检查
- 网络异常，请重试
- 已连接

### 操作文案
- 新建会话
- 发送
- 停止
- 重试
- 切换模型
- 去配置
- 保存并测试
- 返回聊天

### 引导文案
- 连接模型后，即可开始真实对话
- 当前尚未配置可用模型服务
- 配置成功，现在可以开始聊天了

## 导航规则
- App 启动后，仍优先遵循 P1 的会话判断逻辑
- 不因未配置模型而阻止用户进入会话列表或聊天页
- 会话列表页不承担模型切换或配置入口职责
- 只有在用户真正要发起真实请求时，才需要进入模型配置引导
- 配置完成后，应优先返回触发配置前的会话上下文
- 不引入复杂多级设置导航

## 验收关注点
- 在模型已配置时，聊天体验是否与 P1 一致且更真实
- 在模型未配置时，用户是否能快速理解阻塞原因并完成恢复
- 配置引导是否足够轻量，没有把主产品路径变成设置流程
- 鉴权失败、网络失败、服务异常是否能被用户区分感知
- 后续接入第二个 Provider 时，聊天主页面是否无需重新设计

## 本文档不覆盖内容
- 视觉风格与品牌规范
- 具体字段表单设计与安全存储策略
- Provider 接口协议、DTO 与错误码映射
- 技术层 Provider 抽象与模块拆分方案
