# Android 0-1 通用开发规范

## 技术基线
- UI 与交互统一使用 Jetpack Compose + Navigation。
- 开发语言统一使用 Kotlin。
- 架构统一采用 MVI，分层固定为 `View -> ViewModel -> Repository -> DataSource`。
- 跨模块依赖统一通过 Koin 注入，禁止手动跨模块直接 new 具体实现。

## 组件化要求
- 项目采用组件化架构，按功能或领域拆模块，模块边界清晰，避免“公共大杂烩”。
- 模块间仅依赖稳定契约：UI state / intent / repository interface / model，禁止越层调用。
- 每个模块必须维护自己的 README，内容只保留：模块职责、核心类、架构图、数据流。
- 项目根 README 必须说明整体架构、模块关系、核心数据流。
- README 必须高度凝练；模块代码改动涉及职责、核心类、数据流、依赖关系时，同步更新 README。

## 分层约束
- View：只负责 Compose UI、状态消费、事件上抛、导航触发，不写业务逻辑。
- ViewModel：负责 intent 处理、state/reducer、异步调度、错误收敛，不直接访问平台细节。
- Repository：负责聚合领域逻辑与数据来源，对上暴露稳定接口。
- DataSource：负责本地/网络/缓存/系统能力访问，不承载页面业务语义。
- 必须明确 DTO / Entity / UI Model 的转换边界，禁止跨层直接复用同一数据模型。
- 状态单向流动：`Intent -> ViewModel -> State -> View`；副作用必须显式建模。

## 开发执行规范
- 必须先拆子任务，再按子任务逐步执行；当前一步未完成且未检查通过，不进入下一步。
- 每完成一个子任务，都要检查新增或改动文件是否过长、职责是否混乱、方法是否过于复杂。
- 若发现文件或类明显过长、过重、过于复杂，先暂停并询问是否立即重构，再继续下一步。
- 完成后先逐个 review 本次改动到的类和方法，检查语义错误、命名歧义、明显边界 case 漏判。
- 再从交互链路和数据流链路做整体 review，确认无明显状态错误、重复请求、脏数据、导航问题。
- 以上检查完成后，先同步结果并等待回复，再进行下一步开发。

## 质量底线
- 每层职责单一，禁止把 UI 状态处理、领域逻辑、数据访问混写进同一类。
- 命名必须表达真实语义，禁止使用 `Manager`、`Helper`、`Util` 兜底命名。
- MVI 的 State、Intent、Effect 必须收敛明确，避免布尔值堆叠和隐式状态。
- 异常、空态、加载态、重试态必须显式处理，不能只覆盖成功路径。
- 新增依赖、模块、公共模型前，先确认是否已有可复用契约，避免过早抽象和重复建设。
